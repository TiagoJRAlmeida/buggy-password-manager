2. PWM

2.1. Description

pwm is a rudimentary (and insecure!) password file manager, with a format somewhat similar to but simpler than the /etc/shadow file on Linux systems. An example password file is as follows:

admin:68ce6c40:f699d9208839b227a050ce73c8558662
steve:d28d18a9:3b7ac81dfd7bcb735cd0bc596f821afc
linus:02279492:d57aef52733f7ea2fa9f08cd7c44c0ed
edrdo:64507481:8ba7a666b19c26b302db038d093eaae7

Each line has 3 fields:

  • the user ID;
  • a 4-byte salt obtained from /dev/urandom;
  • and an MD5 checksum generated from the salt and the user’s password.

The admin user that appears on the first line (in principle!) is special: subsequent openings of the file require confirmation of this user’s password, and this user cannot be deleted (supposedly!).

2.2. The code

The source code is located in the top directory.

Before you compile it, make sure the BSD library functions are installed:

sudo apt install libbsd-dev

Then, to compile the code, run:

make clean all

The code is compiled by GCC in debug mode, allowing you to use the gdb debugger if desired, and instrumented for fault detection via the AddressSanitizer sanitizer.

The code is organized into several files:

File Description
pwm.h header file with type and function declarations
pwm.c program’s entry point (the main function)
commands.c handling of commands entered by the user
core.c core functions for password file manipulation
validation.c validation functions for checking valid usernames and passwords accepted by the system
utils.c utility functions such as input handling, salt generation, and MD5 hash computation for passwords
md5.c RSA code for MD5 checksum computation, slightly adapted

A small set of C library functions is used. The main ones used are for:

  • string manipulation: strcmp, strcpy, strchr, …;
  • byte sequence copy/comparison: memcpy, memcmp;
  • dynamic memory allocation: malloc, free, strdup;
  • I/O: gets, fgets, fopen, fclose, fprintf, … .

2.3. Introduced bugs and “safe version”

A number of faults (source code defects) have been deliberately introduced into pwm, some of which constitute security vulnerabilities, including:

  • logical programming errors in the validation of usernames and passwords, to be identified through unit tests (see separate section);
  • use of insecure C library functions;
  • improper heap memory management leading to use-after-free, double-free, and memory leak cases;
  • other logical programming errors.

A “safe version” without deliberately introduced faults is the binary program pwmsafe. It is provided without debug information and you may run it in Google Cloud Shell to understand what is the expected behavior of pwm.

3. Detecting and fixing bugs

The aim is to try to uncover and fix bugs in different ways:

  1. We first make use of unit tests to uncover bugs in rules for user name and password validation (pwm/validation.c file).
  2. We then deal with issues reported by the clang static analyzer.
  3. We finally make use of fuzzing to uncover more bugs.

3.1. Unit tests for user/password validation

The code

See pwm/validation.c for the validation code in the functions pwm_is_valid_user and pwm_is_valid_password.

The rules

A username should be considered valid by pwm_is_valid_user if:

  1. It is between 4 and 10 characters long.
  2. It consists only of lowercase letters: 'a' to 'z'.

A password should be considered valid by pwm_is_valid_password if (the constraints are not particularly strong!):

  1. It is between 6 and 12 characters long.
  2. It contains at least one lowercase letter — a to z.
  3. It contains at least one uppercase letter — A to Z.
  4. It contains at least one digit — 0 to 9.
  5. It contains at most one of the following punctuation characters: . , : ! ?.
  6. It does not contain any subsequence found in a (small) blacklist (see source code).

4. Additional challenges

4.1. Design vulnerabilities

We are considering a very simple and vulnerable password manager. Consider the following problems:

  1. Note that an adversary (assuming they have write access to the password file) could remove or alter the position of the admin user’s entry in a password file without pwm noticing. Explain why and how an adversary could exploit this vulnerability. Modify the pwm_open function to fix this design flaw.

  2. Similarly, an adversary with limited technical knowledge could duplicate a line in the file and change only the username to obtain a password equivalent to that of the original user (who may have a known password). This happens because the MD5 checksum only takes into account the salt and the password, but not the username. Generalize pwm_hash_password in utils.c to address this, and modify the program’s logic at other points as needed.

  3. Even after fixing the issues above, nothing prevents arbitrary tampering with the file in order to freely manipulate users and passwords, or to corrupt it. Do you have any proposals for addressing this problem? Explain in the report the general principles of what you propose. You are not required to implement your proposal (though you can also do it!).

4.2. Stack smashing challenge

Preserve or restore the gets calls and the format string vulnerability detected by the static analysis (in 3.2). Explain how these vulnerabilities can be exploited in stack-smashing attacks to improperly obtain a shell. Additionally, you should try to implement the attacks concretely.

For this purpose, you may find several files and utilities in the pwm/ss directory similar to those used in the lab class.

  • To compile pwm and other auxiliary programs in the ss directory, use:

    make clean all [OPTIONS]

where the optional OPTIONS arguments can enable different types of protection for the pwm executable, all disabled by default:

  • CANARIES=1: enables the use of stack canaries;

  • NOXSTACK=1: disables executable code on the stack;

  • PIE=1: enables PIE code;

  • In a Linux machine, you should also disable ASLR, for instance using

echo 0 | sudo tee /proc/sys/kernel/randomize_va_space

In case you have no root access to do that, use the nr.sh script to disable ASLR, for example:

./nr.sh ./pwm
  • A simple hello program is also provided in case you wish to test stack-smashing.
